# 内部処理 (Internal Processing)

このソフトウェアのコアな処理は、Raspberry Pi Picoのメインループによるシリアル通信の定期的なポーリングと、それに伴う状態遷移によって構成されています。開発者が内部の動作を理解しやすくするため、主要なコンポーネントと処理の流れについて解説します。

## ファイル構成

- **`main.cpp`**
  プログラムのエントリポイントです。UART（シリアル通信）やUSBデバイス（TinyUSB）の初期化を行い、無限ループ (`while (1)`) 内でTinyUSBのタスク実行やシリアルデータの受信判定を行います。シリアル通信から1行分のコマンドを受信すると、それを文字列バッファに格納し、パース用の関数を呼び出します。
- **`PokeControllerForPico_Func.h` / `PokeControllerForPico_Func.cpp`**
  実際にコマンドを解釈し、Nintendo Switchへ送信するUSBレポート（コントローラー入力情報）の形態に変換するロジックが含まれています。また、反復入力などの組み込みマクロ機能の状態管理も行います。

## メインループの処理の流れ

`main.cpp` の大まかなフローは以下の通りです。

1. **初期化**
   `uart_init()` でシリアルポートを設定し、`Controller_Init()` でUSBコントローラーのふりをするための `tud_init()` を呼び出します。
2. **タスク実行 (`tud_task()`)**
   TinyUSBのバックグラウンド処理を実行します。これはUSB接続を維持するために不可避の処理です。
3. **シリアル通信の受信 (`uart_is_readable`)**
   PCなどの外部ソフトウェアからUART経由でデータが送られてきていないか確認します。送られてきている場合は1文字ずつ読み込み、改行コード (`\n`) が来るまでバッファ `pc_report_str` に連結します。
4. **コマンドのパース (`ParseLine`)**
   1行分のデータが完成すると `ParseLine()` が呼び出され、内容を解析します（例：コントローラー入力、特定のマクロコマンドなど）。パース後、プログラムの状態変数 `ProgState` が書き換わります。
5. **状態に応じた実行 (`switch(ProgState)`)**
   `ProgState` が実行状態 (`STATE1`) になっていれば、`SwitchFunction()` が呼び出され、解析結果に基づいた実際の操作（Switchへのデータ送信など）が行われます。

## 状態遷移

コード内では、主に2つの状態（State）を組み合わせて動作を制御しています。

### 1. プログラムの進行状態 (`ProgState`)

大きなプログラムのフェーズを管理します（`LOOPSTATE` 列挙型）。

- **`STATE0`** : 待機状態。シリアル通信からの入力を待ち受けている、あるいは1つの処理が完了し次の命令を待機している状態です。
- **`STATE1`** : 実行状態。`ParseLine()` による解析が完了し、PCからの要求や独自マクロコマンドを実際に処理・送信する必要がある状態です。
- **`STATE2`** : 現在の用途としては未使用ですが、将来的な判定・分岐用途などで定義されています。

### 2. 処理の種類 (`proc_state`)

`ParseLine()` の結果、具体的に「何を行うか」を決定する状態変数です（`Proc_State_t` 列挙型）。`SwitchFunction()` はこの変数の値に基づいて動作を切り替えます。

- **`PC_CALL` 系**
  - `PC_CALL` : PCから直接、標準のコントローラー入力フォーマットデータが送られてきた場合。
  - `PC_CALL_STRING` / `PC_CALL_KEYBOARD` 等 : キーボード入力コマンドが送られてきた場合。
- **MCUコマンド系 (例: `MASH_A`, `CHANGETHEDATE` など)**
  組み込みマクロが呼び出された状態です。この状態になると、独自の関数（`GetNextReportFromCommands()` など）が呼び出され、配列で定義された複数ステップの操作が時間をかけて順次実行されます。

1つの「PC_CALL」が完了するか、MCUコマンドが開始されると、`ProgState` は適切に `STATE0` や他の状態に遷移します（MCUコマンド動作中は、タイムアウトなどの時間依存ロジックを回すため、継続的に実行されます）。

## シリアル通信のパース実装 (開発者向け詳細)

外部ソフトウェアと通信する際、Pico側には効率よくデータを処理するためのいくつかの実装依存の工夫が施されています。フォーマット仕様については `docs/serial_protocol.md` を参照してください。

### 1. `0xNNNN` フォーマットのパースとオーバーフローの活用
PC（Pythonスクリプト等）から送られる16進数の文字列は `0x001B` のように `0x` 文字列が付与され、全体が特定の文字数になるようゼロ拡張される仕様になっています。
`ParseLine` ではこの先頭の `0x` を特別な例外処理を用いてスキップするのではなく、**あえて全体を16進数文字として加算し続ける**実装になっています。
ボタンデータを格納する変数 `p_btns` を16ビット符号なし整数（`uint16_t`）で定義することで、先頭の `0` や `x` に起因して想定より大きくなった上位ビットの桁が**オーバーフローによって自然と切り捨てられます**。結果として有効な下位16ビット（`001B`）のみがメモリに安全に残るという仕組みです。

### 2. スティックフラグと2ビット右シフト
プロトコルの仕様上、16ビットデータの最下位2ビットは「スティックを更新するか否かのフラグ」として扱われています。
そのため、Aボタンなどのボタンフラグはプロトコル上では本質的な値より2ビット上にずれた配置で送信されます。`ParseLine` 内ではこれを受信しスティック情報をパースした後、**`p_btns >>= 2;`** として右に2ビットシフトすることで、内部で定義されている本来のEnum値（`SWITCH_A = 0x0004` など）に合致するようデータ構造を復元しています。
